## Package tier model

Three tiers. This is the layout; it is also documented in the architecture page.

1. **Core Platform** — `internal/platform/*`
   Domain, contracts, application services. Fully trusted, compiled in, defines the rules everything else follows.

2. **Built-in module** — `internal/modules/*`
   Infrastructure that implements Platform contracts using the same registration and manifest shape a future external Module would use, but compiled in, required, and fully trusted — not sandboxed, not independently versioned, not optional.
   Postgres is the first example: it lives at `internal/modules/postgres/` (**not** `internal/adapters/postgres/`), registered through `internal/composition/builtin/` the same way an external Module would be discovered.

3. **External module** (future)
   Product/domain capability packs — anime, manga, etc. Governed, independently versioned, discovered at runtime. **Not** part of this repo's initial scaffold.

`internal/adapters/` is reserved for things that are **not** module-shaped — helpers that don't implement a full contract surface on their own (filesystem utilities, crypto helpers). Do not put Postgres there.

Modules compile into the binary, so there is no runtime boundary between Platform and Module code and no in-process sandbox. That is a deliberate trade of isolation for speed (ADR 0007). Treat module trust as something established before the build, not enforced at runtime.

## Non-negotiable rules

- **Dependency direction**: dependencies point inward. Transport → Application services → Contracts/Domain. Adapters/Modules → Contracts → External systems. Domain must never import transport, adapter, or database packages. Application services may depend on Platform contracts, never on concrete Postgres (or other module) types.
- **Error categories**: every contract error maps to one of `InvalidArgument`, `Unauthenticated`, `PermissionDenied`, `NotFound`, `Conflict`, `Unavailable`, `Internal`. Adapters/modules may keep driver-specific errors internally; application services and transports must only ever see these Platform categories.
- **Command handler order**: every command handler follows the same sequence — validate command shape → authenticate caller → authorize via policy → open a `UnitOfWork` → load state through contracts → apply domain rules → persist state and outbox events in the same transaction → return a Platform result type.
- **GraphQL resolvers call services only**: GraphQL is a transport and projection surface, not a persistence layer. Resolvers must call application command or query services — never open a database connection or query Postgres (or any module) directly.
- **Config reload classes**: every configuration field declares a reload class — `Hot` (applies without restart), `Restart` (requires process restart), `Generation` (requires Supervisor to activate a new Generation), `Recovery` (applies only through recovery flow). Classify new config fields before adding them.

## Transaction contract shape (ADR 0001)

The closed `Tx` interface (`Users()/Sessions()/Permissions()/Config()/Outbox()/Credentials()`) gave a new capability no way to join a transaction without editing Core Platform's `Tx` on its behalf — which would mean the Platform requires modification to support a new capability, and that built-in and module-delivered capabilities are not equals. [ADR 0001](https://github.com/mosaic-media/mosaic-architecture/blob/main/docs/adr/0001-transactional-store-extensibility.md) resolves it:

- `Tx` becomes a sealed, opaque transaction-scope marker with no accessor methods.
- Every store — Core Platform or capability — is resolved through a package-level generic function, `Store[T any](tx Tx) (T, error)`, not a named method. Chosen over an extension registry, which loses type safety at the call site, and per-capability `UnitOfWork`, which cannot guarantee atomicity.
- Storage itself becomes a port: a `StorageAdapter` provides the `UnitOfWork` and binds each resolved store to the live transaction, so PostgreSQL becomes a swappable module rather than a privileged implementation.
- Promoting the proven contracts into `contracts/platform/v1` is the *first step* of the Reference capability slice, not deferred to SDK readiness.

**The additive half of this resolution has now landed in code; the subtractive/migration half has not.** As of the "Core contracts rework" slice below, `internal/platform/contracts/` carries `Store[T any](tx Tx) (T, error)` and the `StorageAdapter` port. But `internal/platform/contracts/unit_of_work.go` still declares `Tx` **with all six accessors** (uniform resolution currently delegates to them and is valid only while they survive — sealing `Tx` is a later slice), `contracts/platform/v1` is still empty but for `doc.go`, and every command handler that opens a transaction still calls `tx.Foo()` (no handler moved onto `Store[T]` yet). ADR 0001 rated this high blast radius — every command handler moves off `tx.Foo()`. The remaining migration touches:

- **Needs rework:** Core contracts — the **additive** part (`Store[T]` + `StorageAdapter`) is done ("Core contracts rework" slice); **still outstanding** is sealing `Tx` into an opaque marker (removing the six accessors from `internal/platform/contracts/unit_of_work.go`) and repointing `resolveStore` off the accessors onto the `StorageAdapter` binding. The **PostgreSQL adapter is now done** too ("PostgreSQL adapter rework" slice): a `postgres.StorageAdapter` implements the port, `outbox_atomicity_test.go`/`outbox_worker_test.go` and the `test/contract/suite.go` atomicity proof are migrated onto `Store[T]`, and `Store[T]` is proven transaction-equivalent to the accessors against real PostgreSQL — the postgres `tx` struct itself needed no change, since `Store[T]` delegates to its accessors. **Still outstanding:** the seven command handlers that call `tx.Foo()` (`internal/platform/app/create_local_user.go`, `authenticate_local_user.go`, `revoke_session.go`, `set_user_status.go`, `draft_config_version.go`, `validate_config_version.go`, `activate_config_version.go`), and `internal/transport/graphql/fakes_test.go`'s fake `Tx`. (`test/contract/suite.go` keeps both accessor- and `Store[T]`-based proofs by design, so it is not "moved off" the accessors until sealing removes them.)
- **Unaffected:** Repository scaffold; In-process Event Bus (`internal/platform/events/` — the worker and bus take a directly-injected `EventOutbox`/`EventPublisher`, never a `Tx`); Secret broker (`SecretBroker` isn't reached through `Tx`); GraphQL and Diagnostics/health/Supervisor handoff resolver and service code itself (resolvers only ever call `app.Service`, never `Tx` — only the GraphQL package's test fakes need updating); every query-only app handler (`get_user_by_id.go`, `list_users.go`, `get_active_config_version.go`, `get_config_version.go`, `get_roles_for_user.go`, `get_grants_for_user.go`, `get_effective_permissions.go`), which read through a directly-injected store and bypass `Tx` entirely.

The Reference capability slice cannot be meaningfully re-attempted until this rework lands and `contracts/platform/v1` is actually populated in code — the design is resolved, the implementation is not.

- [x] Core contracts rework (uniform store resolution — MAD-001, first rework slice) — landed the **additive** half of the MAD-001 transaction-shape correction in Core Platform code: `internal/platform/contracts/` now carries `func Store[T any](tx Tx) (T, error)` (`store.go`) and the `StorageAdapter` driven port (`storage_adapter.go`), the uniform, type-safe store-resolution surface MEG-015 §03's Contract Shape shows. **`Tx`'s six accessors (`Users()`/`Sessions()`/`Permissions()`/`Config()`/`Outbox()`/`Credentials()`) are deliberately left intact — sealing `Tx` into an opaque marker is a later rework slice's job, not this one.** Diff is confined to `internal/platform/contracts/` (two new files + `contracts_test.go`); nothing in `internal/modules/postgres/`, `internal/platform/app/`, or `internal/transport/graphql/` changed, and `contracts/platform/v1` is untouched (still `doc.go` only — its promotion is a later slice). **Exact shape chosen (later slices depend on reading this, not guessing):**
  - `StorageAdapter` is `interface { UnitOfWork() UnitOfWork }` — the driven port an engine (built-in PostgreSQL today, a future SQLite drop-in) implements to provide the transaction boundary and bind resolved stores to the live transaction. Only `UnitOfWork()` is exposed; binding is the adapter's internal responsibility, observable through `Store`, not a caller-invoked method. No adapter implements it yet (that is the PostgreSQL-adapter rework slice).
  - `Store[T](tx)` delegates to an **internal, unexported** resolver `resolveStore(tx Tx, typ any) (any, bool)`. The type parameter is carried into the resolver as a typed nil pointer `(*T)(nil)` — because a store contract `T` is an interface type, `var zero T` is a nil interface with no dynamic type, so a switch on `any(zero)` would match nothing; `(*T)(nil)` carries `*T` (`*UserStore`, `*EventOutbox`, …) as its dynamic type instead. `resolveStore` type-switches on that pointer type and, for each of the six known store contracts, **delegates to the matching `Tx` accessor** (`tx.Users()`, `tx.Outbox()`, …). This guarantees `Store[UserStore](tx)` returns the *identical instance* `tx.Users()` would for the same `tx` — the equivalence the test proves — and is a deliberate first-cut valid only while the accessors survive. An unknown `T` returns an `Internal`-category Platform `Error` (fail-loud), never a nil store; nothing at the caller-facing boundary is stringly-typed or requires a store to "ask permission" to join the transaction (MAD-001 §03 rejected option (a), the `any`-keyed extension registry, for losing exactly that — an internal, unexported switch keeps the property). **Migration note for the PostgreSQL-adapter / sealing slices:** `resolveStore` is the single place to repoint when `Tx` is sealed and its accessors removed — swap the accessor delegation for the `StorageAdapter`'s live-transaction binding; `Store`'s signature and every call site stay byte-for-byte unchanged. Command handlers moving from `tx.Foo()` to `Store[Foo](tx)` is a separate later slice (application-service rework) and is untouched here.
  - Test: `contracts_test.go` extends `mockTx` (not replaced) to hold one pointer-backed instance of each mock store as a field (`newMockTx()` constructs them), so both the accessors and `Store` hand back the identical instance — provable by pointer identity rather than empty-struct value equality. `TestStoreResolvesIdenticalInstanceAsNamedAccessor` proves `Store[T](tx) == tx.Foo()` for three store types (User/EventOutbox/Credential); `TestStoreFailsLoudlyForUnboundType` proves an unbound `T` (`SecretBroker`) yields an `Internal` error and a nil store.
